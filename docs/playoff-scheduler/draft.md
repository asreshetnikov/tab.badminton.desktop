Мы автоматизируем процесс создания расписания для турнира по ракетному виду спорта (например, бадминтон), где игрок в рамках одного турнира может принимать участие в трех разных категориях: single, doubles, mixed-doubles. Например, один человек может играть во всех трёх категориях: Men Single, Men Doubles, Mix Doubles. И все категории могут стартовать в один день.

Задача - создать приложение, автоматизизирующее создания расписания.
Есть базовые ограничения:
- у нас есть N-площадок для проведения матчей. Это означает, что в один момент времени может проходить не более N-матчей.
- игрок не может принимать участие в нескольких играх одновременно
- матчи на выбывания (play-off) имеют структуру бинарного дерева: участники последующего матча определяются после завершения предудущих, игры в категории завершаются финалом. 
- формат плей-офф — только single elimination. схема турнира round-robin для групповой стадии - предмет следующих итераций.
- есть время начала матча в каждый игровой день. при этом время окончания не фиксировано.
- на игры каждого этапа плей-офф закладывается временной интервал, позволяющий строить временные слоты. Время может быть разным на разные этапы, однако эти настройки сквозные для всех категорий.

Дело в том, что мы не знаем, кто из игроков выйдет в следующий этап плей-офф, и не можем составить расписание сразу на весь турнир. Однако важно по ходу турнира, завершения матча и продвижения игроков строить расписание с учетом следующих ограничений:
- каждый матч должен иметь время фактического начала и окончания
- игрок должен иметь возможность на отдых между играми. Это означает, что между двумя последовательными играми должно быть не меньше M-минут. Это опция работает на игрока любого пола из любой категории и задается на уровне всего турнира. Это soft constraint, который должна учитывать состема при автоматизированном составлении расписания, должна предупреждать пользователя, если он проектирует расписание в ручном режиму, но не запрещать пользователю назначить время матча.
- каждый матч должен иметь вычисляемое время "not_before_soft" ввиду вышестоящего условие необходимости отдыха игроку между играми.
- также каждые матч должен иметь жестко заданное пользователем условие "not_before_hard". Побеждает макмимальное из двух "not_before_...."
- В матче doubles участвуют 2 игрока. Если у одного из них not_before = 15:00, а у другого = 15:30 — матч назначается не ранее 15:30 (максимум по обоим игрокам)
- Раунды могут перемежаться между категориями. Например: 1/4 финала singles идёт параллельно с 1/8 doubles, если это даёт выигрыш по времени.
- Дни задаются через указание not_before_hard вручную. Последующие матчи, связанные по бинарному дереву, автоматически должны играться позже. 

Ожидаем, что 
- расписание матчей будет выглядеть в виде упорядоченного по приоритет списка матчей, из которого будет браться самый верхний матч и назначаться на конкретную игровую площадку по мере её освобождения.
- последовательность матчей с известными участниками система определит самостоятельно для первых матчей в каждой категории, где участники известны заранее
- по мере движения участников по турнирной сетке система сама будет формировать очередь матчей с учетом органичений и критериев хорошего расписания 
- система будет актуализировать расписание или по инициативе пользователя перед началом турнира, например, когда внесены все необходимые данные, или автоматически по мере внесения результатов матчей.
- при спорных ситуациях в приортет те матчи, которые "разблокируют" следующие быстрее. Для оценки количество зависимых матчей стоит учесть как матчи в дереве данной категории, так и тот факт, что у одного игрока будут несыгранные матчи в разных категориях.
- если для всех матчей указано not_before = 14:30, а площадка освободилась в 14:00 — площадка будет простаивает до 14:30, если пользователь сам не решит иначе
- даты турнира четко определены, если не указано иное через not_before, то матчи играются сегодня, ожидая освобождения площадки
- критерий хорошего расписания - составить и поддерживать очередность матчей таким образом, чтобы минимизировать время окончания матчей турнира с учетом заданных ограничений и заданных not_before
- ожидаемый масштаб турниров - до 128 игроков каждого пола
