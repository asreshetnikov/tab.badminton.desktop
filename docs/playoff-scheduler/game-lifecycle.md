
## Состояния матча:

- создана сетка матчей (бинарное дерево), матч создан, есть связь со следующим матчей, нет time slot, нет игроков, корт не известен.
- автоматический планировщик расписания определяет для матча time slot (предварительное время начала матча). В одном time slot находятся N-матчей, где N - количество игровых кортов на турнире. Игроки по-прежнему не известны
- матчи, игроки в котором известны, встает в лист ожидания, где матчи упорядочены по time-slot (более ранние выше), внутри своего time-slot матчи ранжируются согласно приоритетам: not_before (soft), вес матча (количество зависимых от него матчей)
- первые матчи дня пользователь сам назначает на корты. Матч получает статус Live, фиксируется фактическое время начала матча.
- при завершении матча фиксируется результат, матч получает статус Finished, победитель двигается по сетке дальше, корт освобождается
- при освобождении корта на него автоматически назначается матч самый первый в листе ожидания. Пользователь подтверждает назначение. Пользователь может указать иной матч на своё усмотрение.


 ---
Что реализовано и работает

Пункт 1 — Создание сетки
Реализовано полностью. generateBracket в playoff.service.ts строит бинарное дерево с win_match_id, left_match_id, right_match_id. При создании scheduled_at, court_id, team1_id, team2_id — всё null.

Пункт 2 — Автоматический планировщик (частично)
autoSchedule в scheduler.service.ts существует, назначает scheduled_at и court_id. Концепция приоритетной очереди реализована (priority = category_depth + cross_pending), not_before_soft вычисляется.
Кнопка "Auto-schedule" в UI работает.

Но: планировщик работает только с матчами, у которых уже известны оба игрока (team1_id && team2_id, строка 554). Назначение time slots для матчей с неизвестными игроками не предусмотрено — документ
описывает именно этот сценарий.

Пункт 3 — Лист ожидания (частично)
listUnscheduled возвращает матчи без scheduled_at, они отображаются в левой колонке Schedule. Приоритизация через buildQueue работает. not_before_soft пересчитывается через onMatchCompleted.

Но: нет автоматической реакции на появление игроков в матче — когда advanceWinner записывает команды в следующий матч, планировщик сам не срабатывает. Пользователь должен вручную нажать "Auto-schedule".

Пункт 5 — Завершение матча
Реализовано. updateResult ставит статус finished, advanceWinner двигает победителя, onMatchCompleted записывает actual_end. Поле actual_start тоже заполняется (match.actual_start ?? match.scheduled_at ??
now).

Что не реализовано

Пункт 4 — Статус Live / фиксация фактического старта
- В UI нет кнопки «Начать матч» / «Start»
- Статус in_progress нигде не выставляется (он есть в enum схемы, но не используется в логике)
- actual_start фиксируется только в момент завершения матча как fallback — а не в момент реального старта на корте

Пункт 6 — Авто-назначение при освобождении корта + подтверждение
- Нет механизма «корт освободился → предложи следующий матч из очереди»
- Нет системы подтверждения назначения пользователем
- Нет возможности выбрать альтернативный матч вместо предложенного системой

  ---
Итого: каркас (схема, планировщик, очередь) есть. Не хватает операционной части — перехода матча в in_progress при старте и реактивного назначения на корт при освобождении.

Проблема

Единственный реальный gap — in_progress. Он присутствует в типе, но не используется нигде в логике:
- Нет кнопки «Начать матч» / «Start»
- actual_start проставляется как fallback в момент завершения, а не старта
- Нельзя отличить «стоит в очереди» от «сейчас играется на корте»

Это важно для пункта 6 документа — авто-назначение при освобождении корта — потому что система не знает, какой корт сейчас занят.

  ---
Нужен ли новый статус?

Нет. Набор scheduled → in_progress → finished / walkover / retired полностью покрывает жизненный цикл из документа. Дополнительный статус типа ready (игроки известны, ждёт корта) не нужен — это
UI-фильтрация через team1_id IS NOT NULL, а не смысловое состояние.
