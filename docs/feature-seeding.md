# Фича: посев участников турнира

## Контекст

В текущей реализации каждая запись в `round_teams` имеет целочисленное поле `seed` (NULL означает отсутствие посева). `generateBracket` сортирует команды по `seed ASC` (NULL в конец) и раскладывает их по слотам сетки через `buildSeedingOrder`.

Этого достаточно для точного посева (seed 1 → слот 1, seed 2 → слот N), но не поддерживает групповой посев — стандартную практику в бадминтоне, когда точное место в сетке для нескольких равных по силе игроков определяется жеребьёвкой.

---

## Цель

Позволить организатору назначить каждой entry в категории один из двух типов посева:

| Тип | Примеры значений | Смысл |
|-----|-----------------|-------|
| **Точный** | 1, 2, 3, 4, … | Конкретная позиция в сетке |
| **Групповой** | 1/2, 3/4, 5/8, 9/16, 17/32 | Диапазон позиций; точное место определяется жеребьёвкой |

Цель посева: **чем сильнее игроки, тем позже они могут встретиться в сетке**.

| Посев | Самая ранняя возможная встреча |
|-------|-------------------------------|
| 1 и 2 | Финал |
| 1/2 и 3/4 | Полуфинал |
| 1/4 и 5/8 | Четвертьфинал |
| 1/8 и 9/16 | 1/8 финала |
| … | … |

---

## Определения

### Групповой посев: нотация `lo/hi`

Запись `lo/hi` означает: entry занимает одну из позиций в диапазоне `[lo, hi]` (включительно).

- `1/2` — первый или второй посев (финалисты по рейтингу)
- `3/4` — третий или четвёртый посев
- `5/8` — пятый–восьмой посев
- `9/16` — девятый–шестнадцатый посев

`hi` всегда равен степени двойки, а `lo = hi/2 + 1`. Это соответствует уровням турнирной сетки размером `hi`.

### Гарантия разделения

Для группового посева `lo/hi`:
- все участники группы размещаются в **разных** секторах сетки размером `size/hi`
- внутри своего сектора точная позиция определяется жеребьёвкой (случайный порядок)

Это гарантирует, что два участника из одной группы не могут встретиться раньше раунда, соответствующего `hi`.

---

## Алгоритм размещения в сетке

### Входные данные

- `n` — количество участников в раунде
- `size = nextPowerOf2(n)` — размер сетки
- Список `entries`, каждая с полем `seed`:
  - `null` — без посева
  - целое `k` — точный посев k
  - дробное `lo/hi` — групповой посев

> **Сетка без посева** — валидный и распространённый случай: все entries имеют `seed = null`. В этом случае шаги 1 и 2 пропускаются, все участники попадают в unseeded-пул и размещаются полностью случайно через `Math.random()`. `resolveDraw` в этом сценарии просто перемешивает весь список.

### Шаги

**1. Разбиение на группы**

Сгруппировать entries по типу посева:
```
exact[k]   = entry с точным посевом k
group[lo/hi] = список entries с групповым посевом lo/hi
unseeded   = entries без посева
```

**2. Жеребьёвка внутри групп**

Для каждого группового посева `lo/hi`: перемешать entries случайно → назначить им целочисленные позиции `lo, lo+1, …, hi` (или до `min(hi, n)` если участников меньше).

`resolveDraw` использует **чистый `Math.random()`** (Fisher-Yates shuffle) без какой-либо детерминированной логики — каждая жеребьёвка даёт равновероятный результат.

После этого шага у каждой entry есть конкретный целочисленный номер.

**3. Распределение unseeded**

Оставшиеся слоты (не занятые посеянными) заполнить unseeded entries в случайном порядке (`Math.random()`, Fisher-Yates).

**4. Размещение по `buildSeedingOrder`**

Применить существующий алгоритм `buildSeedingOrder(size)` как сейчас — он гарантирует, что:
- seed 1 и seed 2 — в разных половинах → встреча только в финале
- seed 3 и seed 4 — в разных четвертях → встреча не раньше полуфинала
- и т.д.

Группы, сформированные на шаге 2, уже обеспечивают нужное разделение, т.к. их члены получили разные позиции из диапазона `lo..hi`.

## Хранение данных

Посев хранится на двух уровнях:

**Объявленный посев** (`tournament_teams.seed_lo`, `tournament_teams.seed_hi`) — задаётся организатором на уровне категории и един для всех раундов этой категории.

```sql
ALTER TABLE tournament_teams ADD COLUMN seed_lo INTEGER;
ALTER TABLE tournament_teams ADD COLUMN seed_hi INTEGER;
```

- Точный посев k: `seed_lo = k`, `seed_hi = NULL`
- Групповой посев `lo/hi`: `seed_lo = lo`, `seed_hi = hi`
- Без посева: `seed_lo = NULL`, `seed_hi = NULL`

**Разыгранный посев** (`round_teams.seed`) — конкретная целочисленная позиция после жеребьёвки. Хранится на уровне раунда, поскольку разные раунды одной категории могут иметь разный размер сетки. Именно это поле использует `generateBracket`.

Разделение позволяет:
- переигрывать жеребьёвку без потери исходных данных
- показывать в UI «посеян как 5/8, место в сетке — 7»

### Жеребьёвка

`resolveDraw(roundId)` читает `seed_lo`/`seed_hi` из `tournament_teams` через JOIN и записывает результат в `round_teams.seed`. Изменение объявленного посева через `tournament_teams:setSeed` сбрасывает `round_teams.seed` во всех раундах категории и блокируется, если в любом раунде категории есть матчи.

## UI

### Ввод посева (TournamentTeams / Entries категории)

Посев задаётся на экране Entries категории (TournamentTeams), а не в GroupsView — потому что посев единый для всех раундов категории.

- Переключатель режима: **Exact** (точный) / **Group** (групповой)
- В режиме Exact: числовое поле для каждой entry (1, 2, 3, …)
- В режиме Group: выбор группы (1/2, 3/4, 5/8, 9/16, 17/32) для каждой entry
- Переключение режима сбрасывает все объявленные посевы (с подтверждением)
- Заблокировано, если в любом раунде категории есть матчи

В GroupsView посев отображается только для чтения рядом с именем участника.

### Кнопка «Draw seedings» (Провести жеребьёвку)

- **Активна всегда**, если в раунде нет сгенерированных матчей — независимо от наличия посевов.
  - При наличии групповых посевов — перемешивает участников внутри каждой группы и назначает им конкретные позиции.
  - Для unseeded участников — назначает случайные позиции из оставшихся слотов.
  - При полностью unseeded сетке — перемешивает всех участников целиком.
- Результат записывается в `round_teams.seed` (resolved integer).
- Показывает результат жеребьёвки в модальном окне перед подтверждением (можно отменить и провести заново).
- **Неактивна** (disabled), если в раунде уже есть матчи — требуется сначала Regenerate.

### Кнопка «Generate Bracket»

`generateBracket` **требует, чтобы у всех участников был resolved seed** (поле `seed` заполнено). Если хотя бы у одного участника `seed = null`:

- Generate Bracket **блокируется** в UI (кнопка disabled с подсказкой «Сначала проведите жеребьёвку»).
- `generateBracket` на уровне сервиса тоже проверяет это условие и бросает ошибку, если `seed = null` у кого-либо из участников — защита от обхода через IPC.

Таким образом, **Draw всегда предшествует Generate Bracket** — явно, по действию пользователя. Автоматический внутренний draw внутри `generateBracket` не делается: это скрыло бы жеребьёвку от пользователя и лишило бы его возможности увидеть и подтвердить результат.

### Изменение посева и повторная жеребьёвка

После генерации сетки (`generateBracket` создал матчи) **запрещено**:
- изменять `seed_lo` / `seed_hi` (`setSeed` бросает ошибку если матчи существуют)
- проводить повторную жеребьёвку (`resolveDraw` и кнопка Draw заблокированы)

Единственный путь пересмотреть посев — нажать **Regenerate** (существующая кнопка), которая удаляет все матчи раунда. После этого Draw и setSeed снова доступны.

Это намеренно консервативное поведение для первой реализации: проще запретить и явно разблокировать через Regenerate, чем обрабатывать кейсы частичного пересева при уже начатых матчах.

### Визуализация в сетке (PlayoffBracket)

- На карточке матча рядом с именем команды показывать посев в скобках: `(1)`, `(3/4)`, `(5/8)`
- Tooltip или раскрытие по клику: «посев 3/4, место в сетке 3»

## Валидация

| Правило | Ошибка |
|---------|--------|
| `lo < hi`, оба > 0 | «Некорректный диапазон посева» |
| `hi` — степень двойки | «hi должен быть степенью двойки (2, 4, 8, 16, …)» |
| `lo = hi/2 + 1` | «lo должен быть hi/2 + 1 (например, 5/8, а не 4/8)» |
| Два участника с одним точным посевом k | «Посев k уже занят» |
| Группа `lo/hi` имеет > `hi - lo + 1` участников | «Слишком много участников для группы lo/hi» |
| `size` сетки < `hi` | «Размер сетки меньше максимального посева» |

## Примеры

### Пример 1: 16 участников с полным посевом

| Посев | Участники | Результат после draw |
|-------|-----------|---------------------|
| 1 | Иванов | seed = 1 |
| 2 | Петров | seed = 2 |
| 3/4 | Сидоров, Козлов | seed = 3 или 4 (жеребьёвка) |
| 5/8 | A, B, C, D | seed = 5, 6, 7, 8 (жеребьёвка) |
| без посева | 8 участников | seed = 9–16 (жеребьёвка) |

### Пример 2: 5 участников, сетка 8

| Посев | Участник |
|-------|---------|
| 1 | A |
| 2 | B |
| без посева | C, D, E |

`buildSeedingOrder(8)` → [1,8,4,5,2,7,3,6]

- Слот 1 → seed 1 → A
- Слот 2 → seed 8 → bye
- Слот 3 → seed 4 → C/D/E жеребьёвка
- Слот 4 → seed 5 → C/D/E жеребьёвка
- Слот 5 → seed 2 → B
- Слот 6 → seed 7 → bye
- Слот 7 → seed 3 → C/D/E жеребьёвка
- Слот 8 → seed 6 → bye

A и B — в разных половинах, встреча только в финале.

## Связь с существующим кодом

| Компонент | Изменение |
|-----------|-----------|
| `schema.ts` | `seed_lo`, `seed_hi` в `tournament_teams`; `round_teams.seed` остаётся как resolved seed |
| `tournament-team.repo.ts` | Метод `setSeed(tournamentTeamId, lo, hi)` (блок если матчи есть в любом раунде категории); сбрасывает `round_teams.seed` во всех раундах |
| `round-team.repo.ts` | Метод `resolveDraw(roundId)`: читает `seed_lo`/`seed_hi` из `tournament_teams` через JOIN, пишет resolved seed в `round_teams.seed`; блок если матчи есть |
| `playoff.service.ts → generateBracket` | Валидация: все `round_teams.seed` должны быть не null; бросает `UNRESOLVED_SEEDINGS` иначе |
| IPC | Новые каналы: `tournamentTeams:setSeed`, `roundTeams:draw` |
| `TournamentTeams` | Ввод и редактирование объявленного посева (режимы Exact/Group) |
| `GroupsView` | Отображение посева (read-only); кнопка Draw; кнопка Generate Bracket (disabled если есть unresolved seeds) |
| `PlayoffBracket` | Показ посева на карточке матча |

## Открытые вопросы

1. ~~**Переигровка жеребьёвки**~~ — **решено**: Draw и setSeed заблокированы при наличии матчей; единственный путь — Regenerate (удалить матчи).
2. ~~**Generate Bracket без draw**~~ — **решено**: Generate Bracket заблокирован, пока у любого участника `seed = null`; пользователь обязан явно нажать Draw перед генерацией сетки.
3. **Импорт посевов**: при импорте игроков (CSV/BWF) стоит ли поддерживать импорт посевов?
4. ~~**Частичный посев**~~ — **решено**: `resolveDraw` обрабатывает смешанный состав — exact, group и unseeded участники корректно сосуществуют; unseeded заполняют оставшиеся слоты случайно.
5. **Отображение в расписании**: показывать ли посев в карточке матча на Schedule screen?
